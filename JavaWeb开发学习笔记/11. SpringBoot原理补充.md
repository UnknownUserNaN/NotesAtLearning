1. 配置优先级：
	SpringBoot支持三类不同的配置文件，即application.properties、application.yml和application.yaml。
	除了以上3种配置文件外，SpringBoot为了增强程序的扩展性，除了支持配置文件的配置方式以外，还支持另外两种常见的配置方式：
	1. Java系统属性配置 （格式： -Dkey=value）
	```YAML
	-Dserver.port=9000
	```
	2. 命令行参数 （格式：--key=value）
	```YAML
	--server.port=10010
	```
	上述五种配置方式的优先级：**命令行参数 > Java系统属性参数 > properties参数 > yml参数 > yaml参数**

3. Bean对象的作用域：
	在前面我们提到的IOC容器（如`@Component`、`@Controller`、`@Service`、`@Repository`）当中，默认bean对象是单例的 (只有一个实例对象)。
	> @Mapper 不是 Spring 原生提供的注解，而是 MyBatis 框架的注解。Spring 和 MyBatis 是两个不同的框架，它们通过整合包（如 mybatis-spring）协同工作，但各自保持独立性。

	在Spring中支持五种作用域，后三种在web环境才生效：
	
	| 作用域         | 说明                        |
	| ----------- | ------------------------- |
	| singleton   | 容器内同名称的bean只有一个实例（单例）（默认） |
	| prototype   | 每次使用该bean时会创建新的实例（非单例）    |
	| request     | 每个请求范围内会创建新的实例（web环境中，了解） |
	| session     | 每个会话范围内会创建新的实例（web环境中，了解） |
	| application | 每个应用范围内会创建新的实例（web环境中，了解） |
	可以借助Spring中的@Scope注解来进行配置作用域：
	```java
	@Scope("prototype")
	@RequestMapping("/dept")
	@RestController
	public class DeptController{
	...
	}
	```
	要在特定的作用域下调用Bean对象，可以参照下面的方式：
	```Java
	@SpringBootTest
	class SpringbootWebConfigApplicationTests {
	
	    @Autowired
	    private ApplicationContext applicationContext; //IOC容器对象
	
	    //bean的作用域
	    @Test
	    public void testScope(){
	        for (int i = 0; i < 10; i++) {
	            DeptController deptController = applicationContext.getBean(DeptController.class);
	            System.out.println(deptController);
	        }
	    }
	}
	```
	- IOC容器中的bean默认使用的作用域：singleton (单例)
		- 默认singleton的bean，在**容器启动时**被创建，可以使用@Lazy注解来延迟初始化(**延迟到第一次使用时**)
	- prototype的bean，每一次使用该bean的时候都会创建一个新的实例。
		- 实际开发当中，绝大部分的bean是单例的，因为通常是不保存数据的无状态的bean使用单例，而Controller，Service，Dao（或Mapper）等bean均属于无状态的bean。
	- Spring容器的Bean是线程安全的吗？
		- bean的线程安全取决于Bean的状态和Bean的作用域；
		- 单例状态下：如果是无状态的Bean，由于Bean内部不会保存任何状态信息，因而是线程安全的；如果是有状态的Bean，Bean内部会保存状态信息，多个线程同时操作该Bean时，会出现数据不一致的问题，因此这样的Bean是线程不安全的（可以通过将Bean设置成非单例prototype来规避这种情况）。
4. 第三方Bean：
	如果某个类它不是我们自己编写的，而是我们引入的第三方依赖当中提供的，那么此时我们是无法使用 `@Component` 及其衍生注解来声明bean的，此时就需要使用`@Bean`注解来声明bean 了。
	- 方法1：在启动类中使用`@Bean`注解声明这个Bean对象。
	- 方法2：若要管理的第三方 bean 对象，建议对这些bean进行集中分类配置，可以通过 `@Configuration` 注解声明一个配置类。如：
	```java
	@Configuration
	public class OSSConfig {
	    @Bean
	    public AliyunOSSOperator aliyunOSSOperator(AliyunOSSProperties ossProperties) {
	        return new AliyunOSSOperator(ossProperties);
	    }
	}
	```
	通过@Bean注解的name 或 value属性可以声明bean的名称，如果不指定，默认bean的名称就是方法名。
	如果第三方bean需要依赖其他bean对象，直接在bean定义方法中设置形参即可，容器会根据类型自动装配。
5. Spring底层原理分析：
	SpringBoot框架之所以使用起来更简单更快捷，是因为SpringBoot框架底层提供了两个非常重要的功能：一个是**起步依赖**，一个是**自动配置**。
	1. 起步依赖：
		通过SpringBoot所提供的起步依赖，就可以大大的简化pom文件当中依赖的配置，从而解决了Spring框架当中依赖配置繁琐的问题。
		**起步依赖的原理就是Maven的依赖传递**。
	2. 自动配置：
		SpringBoot的自动配置就是当spring容器启动后，一些配置类、bean对象就自动存入到了IOC容器中，不需要我们手动去声明（或者说，使用时可以通过`@Autowired`引入），从而简化了开发，省去了繁琐的配置操作。
		- 引入进来的第三方依赖当中的bean以及配置类不能直接被SpringBoot自动配置。原因是：在类上添加`@Component`注解来声明bean对象时，还需要保证`@Component`注解能被Spring的组件扫描到。
		- 要解决这个问题，有两种方法：
			1. 在启动类上添加注解，通过`@ComponentScan` 指定组件扫描的范围。但是，这种方法往往需要大量扫描，会影响程序运行效率，也不是Spring官方推荐的方案。
				```java
				@SpringBootApplication
				@ComponentScan({"com.itheima","com.example"}) //指定要扫描的包
				public class SpringbootWebConfigApplication {
				    public static void main(String[] args) {
				        SpringApplication.run(SpringbootWebConfigApplication.class, args);
				    }
				}
				```
			2. 在启动类上使用`@Import`注解导入（使用`@Import`导入的普通的类会被Spring加载到IOC容器中）。
				```Java
				@Import(TokenParser.class) //导入的类会被Spring加载到IOC容器中
				@SpringBootApplication
				public class SpringbootWebConfigApplication {
				    public static void main(String[] args) {
				        SpringApplication.run(SpringbootWebConfigApplication.class, args);
				    }
				}
				```
				这种方法当要引入一个第三方依赖时，往往还要知道第三方依赖中有哪些配置类和哪些Bean对象。这往往会比较繁琐。因此，大部分第三方的依赖都会自行提供一个注解（通常形如`@Enablexxx`），在使用时只需在启动类上加上`@EnableXxx`注解即可，比较方便。
				值得一提的是，此种方法在引入普通类、配置类（使用`@Configuration`修饰的类）和ImportSelector实现类（专用于动态引入）时，实际引入效果会有所不同：
				1. **导入普通类**：当你使用`@Import`导入一个普通类时，Spring会把这个类作为一个Bean定义注册到容器中。换句话说，这个类会被实例化并管理起来。例如，如果你有一个简单的POJO类，通过`@Import`导入后，它就会成为一个Spring Bean。
				2. **导入配置类**：配置类通常是指使用了`@Configuration`注解的类。当你导入一个配置类时，Spring会处理这个配置类中所有`@Bean`注解的方法，将这些方法返回的对象注册为Bean。此外，配置类可能还会通过`@ComponentScan`扫描其他组件，或者通过`@Import`导入其他配置。所以，导入配置类可能会注册多个Bean。
				3. **导入ImportSelector接口实现类**：`ImportSelector`接口允许你根据条件动态选择要导入的配置类。实现这个接口的类需要返回一个字符串数组，每个字符串代表一个需要导入的配置类的全限定名。Spring会调用`selectImports`方法，然后根据返回的类名数组进行导入。这种方式常用于基于条件（如类路径下是否存在某个类）的自动配置。
6. SpringBoot自动配置的原理分析：
	源码入口就是 `@SpringBootApplication` 注解，在这个注解中封装了3个注解，分别是：
	- `@SpringBootConfiguration`
	  声明当前类是一个配置类（这也就是为什么我们可以在启动类中声明类对象，从而将其引入程序）
	- `@ComponentScan`
	  进行组件扫描（SpringBoot中默认扫描的是启动类所在的当前包及其子包）
	- `@EnableAutoConfiguration`
	  封装了@Import注解（Import注解中指定了一个ImportSelector接口的实现类）
	  在实现类重写的selectImports()方法，读取当前项目下所有依赖jar包中META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports两个文件里面定义的配置类（配置类中定义了@Bean注解标识的方法）。
	但是，在上面的imports文件中定义的配置类非常多，而且每个配置类中又可以定义很多的bean，那这些bean都会注册到Spring的IOC容器中吗？当然不是。在声明bean对象时，上面有加一个以 `@Conditional` 开头的注解，这种注解的作用就是**按照条件**进行装配，只有满足条件之后，才会将bean注册到Spring的IOC容器中。
	`@Conditional`注解（条件装配注解）：
	- 作用：按照一定的条件进行判断，在满足给定条件后才会注册对应的bean对象到Spring的IOC容器中。
	- 位置：方法、类
	- `@Conditional`本身是一个父注解，派生出大量的子注解：
	  - `@ConditionalOnClass`：判断环境中有对应字节码文件，才注册bean到IOC容器。
		 如 `@ConditionalOnClass(name="io.jsonwebtoken.Jwts")`
	  - `@ConditionalOnMissingBean`：判断环境中没有对应的bean(类型或名称)，才注册bean到IOC容器。
	  - `@ConditionalOnProperty`：判断配置文件中有对应属性和值，才注册bean到IOC容器。
		  如`@ConditionalOnProperty(name ="name",havingValue = "itheima")`
7. Starter：
	starter指的就是SpringBoot当中的起步依赖。
	在SpringBoot当中已经给我们提供了很多的起步依赖了，我们为什么还需要自定义 starter 起步依赖？这是因为在实际的项目开发当中，我们可能会用到很多第三方的技术，并不是所有的第三方的技术官方都给我们提供了与SpringBoot整合的starter起步依赖，但是这些技术又非常的通用，在很多项目组当中都在使用。
	在自定义一个起步依赖starter的时候，按照规范需要定义两个模块：
	1. starter模块（进行依赖管理，把程序开发所需要的依赖都定义在starter起步依赖中）
	2. autoconfigure模块（自动配置）
