1. B/S架构与C/S架构：
	- BS架构：Browser/Server，浏览器/服务器架构模式。客户端只需要浏览器，应用程序的逻辑和数据都存储在服务端。
	    - 优点：维护方便
	    - 缺点：体验一般
	- CS架构：Client/Server，客户端/服务器架构模式。需要单独开发维护客户端。
	    - 优点：体验不错
	    - 缺点：开发维护麻烦
2. 我们在项目中，直接基于**SpringFramework**进行开发，存在两个问题：
	- 配置繁琐
	- 入门难度大
	所以基于此呢，spring官方推荐我们从另外一个项目开始学习，那就是目前最火爆的SpringBoot。 通过springboot就可以快速的帮我们构建应用程序，所以**springboot**呢，最大的特点有两个：
	- 简化配置
	- 快速开发
	**Spring Boot 可以帮助我们非常快速的构建应用程序、简化开发、提高效率 。**
	**而直接基于SpringBoot进行项目构建和开发，不仅是Spring官方推荐的方式，也是现在企业开发的主流。**
3. 实际的后端程序开发中，通常会将程序分为如下三层：
	- Controller：控制层。接收前端发送的请求，对请求进行处理，并响应数据。
	- Service：业务逻辑层。处理具体的业务逻辑。
	- Dao：数据访问层(Data Access Object)，也称为持久层。负责数据访问操作，包括数据的增、删、改、查。

	- Controller层调用Service层来进行逻辑处理（Service层处理完后，把处理结果返回给Controller层）
	- Serivce层调用Dao层（逻辑处理过程中需要用到的一些数据要从Dao层获取）
	- Dao层操作文件中的数据（Dao拿到的数据会返回给Service层）

4. 软件开发中常涉及的两个概念-内聚与耦合：
	- **内聚：**软件中各个功能模块内部的功能联系。
	- **耦合：**衡量软件中各个层/模块之间的依赖、关联的程度。
	- **软件设计原则：高内聚低耦合。**
> 		**高内聚****：**指的是一个模块中各个元素之间的联系的紧密程度，如果各个元素(语句、程序段)之间的联系程度越高，则内聚性越高，即 "高内聚"。
> 		**低耦合：**指的是软件中各个层、模块之间的依赖关联程序越低越好。
5. 我们各层的程序实现里，不应该明确地指定采用的业务对象类型（比如明确地在new对象时指定UserService类型），这引入了“耦合”。为了解耦，我们应当将项目中的类交给IOC容器管理，当需要采用指定的业务对象时，由IOC容器交付。这样，业务程序就对业务对象完全“无感”了，从而实现了解耦。为了降低各个业务层之间的**耦合**，就必须用到以下的两个概念——“控制反转”和“依赖注入“： 
	- **控制反转：** Inversion Of Control，简称**IOC**。对象的创建控制权由程序自身转移到外部（容器），这种思想称为控制反转。
	    - 对象的创建权由程序员主动创建转移到容器(由容器创建、管理对象)。这个容器称为：IOC容器或Spring容器。
	- **依赖注入：** Dependency Injection，简称**DI**。容器为应用程序提供运行时，所依赖的资源，称之为依赖注入。
	    - 程序运行时需要某个资源，此时容器就为其提供这个资源。
	    - 例：EmpController程序运行时需要EmpService对象，Spring容器就为其提供并注入EmpService对象。
	为了实现这一点，通常需要依赖Spring中的Bean对象这个概念：
	- **bean对象：** IOC容器中创建、管理的对象，称之为：bean对象。
	Spring中通常使用两个注解来完成上述两个操作：
	1. **@Component**：加在要接管的类上面（注意是**加在实现类上**，而非接口上），声明这是一个bean对象，表示该类的对象交由IOC容器统一管理。
	2. **@Autowired**：加在要接管的成员变量上面。该成员变量不再使用new方法赋值，而是通过Spring的IOC容器，在程序运行的过程中自动从容器中获取并为成员变量赋值。
6. **@Component注解的衍生注解**：
	- `@Controller`用于标注在控制层类上，`@Service`用于标注在业务层类上，`@Repository`用于标注在数据访问层类上（由于此部分通常与Mybatis整合，故用得比较少）。`@Controller`通常不单独使用，因为通常我们已经通过`@RestController`注解来一同声明了这是个Controller。
	- 通过上述四个注解声明bean的时候，可以通过注解的value属性指定bean的名字。如果没有指定，默认为类名首字母小写（如UserService类会自动命名为userService），这也启发我们必须要遵守Java项目的一般命名规范。
	- 上面这四个注解要想生效，需要被Spring中的`@ConponentScan`扫描到。该注解默认包含在SpringBoot的启动类的注解`@SpringBootApplication`中，默认的扫描范围是**启动类所在包及其子包**。
7. **@Autowired注解的使用方法**：
	- 该注解有三种注入方式：
		1. **属性注入**：
			```java
@RestController
public class UserController {

//方式一: 属性注入
@Autowired
private UserService userService;

}
			```
			- 优点：代码简洁、方便快速开发。
			- 缺点：隐藏了类之间的依赖关系、可能会破坏类的封装性。
		2. **构造函数注入**：
			```java
@RestController
public class UserController {

//方式二: 构造器注入
private final UserService userService; // 为了安全，通常要给该成员变量赋予final关键字，防止被子类修改

@Autowired //如果当前类中只存在一个构造函数, @Autowired可以省略
public UserController(UserService userService) {
	this.userService = userService;
}

}   
			```
			- 优点：能清晰地看到类的依赖关系、提高了代码的安全性。
			- 缺点：代码繁琐、如果构造参数过多，可能会导致构造函数臃肿。
			- 注意：如果只有一个构造函数，@Autowired注解可以省略。（通常来说，也只有一个构造函数）
		3. **setter（set方法）注入**：
			```java
@RestController
public class UserController {

//方式三: setter注入
private UserService userService;

@Autowired
public void setUserService(UserService userService) {
	this.userService = userService;
}

}   
			```
			- 优点：保持了类的封装性，依赖关系更清晰。
			- 缺点：需要额外编写setter方法，增加了代码量。
	- 该注解通常用于提示Spring，在初始化该类对象的某个成员变量时，去IOC容器中寻找对应的类。但是，我们要初始化的成员变量，其类别（接口）可能有多个类的实现，也就是一个接口对应着多个bean对象。此时若不作特殊指定，SpringBoot在启动时会报错，表示找不到合适的bean对象（因为有多个bean对象都符合要求）。此时有以下三种解决方案：
		1. **在声明bean对象的时候使用@Primary注解**，表示这是要默认采用的bean对象；
		2. **在使用@Autowired注解的时候附加@Qualifier("xxx")注解**，声明自己要指定的bean对象（指定实现接口的类）；
		3. **将@Autowired注解更换为@Resource(name="xxx")注解**，声明声明自己要指定的bean对象（指定实现接口的类）。
